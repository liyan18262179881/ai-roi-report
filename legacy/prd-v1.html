00 / OVERVIEW

产品概述

一句话:帮中小企业在投钱做 AI 之前,先算清楚值不值、怎么做、要花多少、多久回本。

这是一个自助式网页工具。企业主或业务负责人打开链接,选择一个业务场景、填入自己公司的真实数字,当场就能看到一份 ROI 测算结果和落地方案,并生成两份可交付报告——一份给老板做决策,一份给乙方(我们)做报价与施工。

要解决的真实问题

中小企业老板都听说过"AI 能提效",但卡在两个地方下不了决心:第一,不知道值不值——投进去的钱多久能赚回来没人给得出数字;第二,不知道找谁靠谱——市面上的 AI 服务商说得天花乱坠,但拿不出针对自己公司的具体测算。这个工具把模糊的"AI 提效"变成一份带具体数字的决策依据。

MVP 要验证的核心假设

核心假设

「一份针对企业真实情况、数字精准的 ROI 评估报告,能不能打动企业、促成它愿意为后续的 AI 落地服务付费?」

注意:MVP 验证的不是"哪个场景最热门",而是"这套质量能不能带来成交"。对一个从零起步、靠信任和案例破局的团队来说,拿到第一个付费客户和第一个真实案例,比覆盖面广更重要。

01 / USERS & SCENES

用户与场景

谁在用、用来做什么决定,以及为什么场景只做一个深、其余做入口。

两类核心用户

用户诉求对应产出
企业老板 / 老总这事值不值得做?投多少、几个人、多久回本?老板版报告
我们(乙方)怎么报价、怎么交付、技术怎么选、智能体怎么搭?乙方版报告

同一套输入,切出两种视角。老板版帮客户做决策、促成成交;乙方版帮我们自己定价、保证利润。这是产品设计上一个聪明的杠杆。

四个场景,为什么只做透一个

四个场景的计算骨架相通(都是「价值 ÷ 投入 = ROI」),但每个场景具体问什么数、用什么公式、调什么 AI、耗多少 token 完全不同——本质是四套独立的「场景引擎」。MVP 阶段不可能四套都做准,所以:

场景MVP 状态说明
电商内容生产
图文 / 详情页 / 素材		做透字段、公式、token 估算都做准。你最懂,参数最有把握。
客服机器人即将上线入口可点,点进去显示「评估中,留资优先体验」
内部知识库即将上线同上
销售助手即将上线同上(注意:此场景是「增收」逻辑,非省钱)
「即将上线」不是占位,是情报采集

通过统计哪个「即将上线」场景被点击 / 留资最多,你能用近乎零成本拿到「哪个场景需求最大」的真实信号,直接指导下一个做哪个。等于用「做透一个」的成本,顺手拿到「做浅四个」的核心情报。

02 / CORE FLOW

核心流程

从打开链接到拿到报告,企业要走的完整路径。

1
落地页
一句话价值主张 + 「1 分钟算出你的 AI 账」按钮。无需注册登录。
2
选择场景
四个场景卡片。电商内容可进入;其余三个标「即将上线」,点击触发留资。
3
填写数据
5–7 个老板能随口说出的数字(人数、薪资、日产量、提升倍数等)。带默认值降低门槛。
4
实时 ROI 测算
填完即算:月省/增收、年度价值、总投入、回本周期。数字随输入实时变化。
5
查看落地方案
三阶段路线图 + 人员配置 + 周期投入。老板看投入回报,乙方看施工细节。
6
生成 / 下载报告 → 留资
在线查看双版本报告;下载 PDF 或获取完整版前留下联系方式(转化点)。
设计要点:门槛前置最低,价值尽早出现

第 3 步是整个漏斗最大的流失点。中小企业老板最怕「让我填一堆我不知道的数字」。所以字段要少、要有默认值、每个字段一句话解释从哪查。理想情况是填到第 2、3 个字段时,结果区就开始显示数字了——让人「上瘾」继续填。

字段克制原则

MVP 阶段宁可少问、用合理默认值,也不要因为追求精度把字段堆到 15 个——填写完成率比测算精度更重要。「精度」可以放到留资后由你人工跟进时补全。

05 / CALCULATION LOGIC

ROI 计算逻辑

电商内容场景的公式定义。所有系数都应可在后台配置。

角度一:省人力成本

可精简人数 = 团队人数 × (1 − 1 / 提升倍数)

月省成本 = 可精简人数 × 人均月薪 × 1.3

举例:8 人团队、提升 3 倍 → 可精简 8×(1−1/3)≈5.3 人;月薪 8000 → 月省 ≈ 5.3×8000×1.3 ≈ 5.5 万。系数 1.3 表示社保公积金等综合用工成本。

角度二:产能翻倍

新日产能 = 日产出 × 提升倍数

等效新增产出 = 日产出 × (提升倍数 − 1) × 30(天)

同样的人,日产能从 80 → 240 件,等于每月白赚约 4800 件产出。这个角度对「想扩张但不想加人」的老板更有吸引力。

Token / API 成本估算

项目单价假设(可调)月成本公式
文案生成~0.02 元 / 件月产量 × 0.02
图片生成~0.15 元 / 张月产量 × 0.15
平台适配变体× 平台数系数基础成本 × (1 + 0.2 ×(平台数−1))

月产量 = 日产出 × 30。单价为示意值,正式版需用真实模型计费(输入/输出 token 分别计价)+ 图片接口报价校准。

06 / DUAL REPORTS

双版本报告

同一套输入,两份产出。边界严格区分:老板不看技术,乙方要看施工。

老板版 看投入回报

目标:帮老板判断「值不值得做」,语言简单、有说服力。

  • 每月可省 / 增收金额(双角度)
  • 总投入与三阶段节奏
  • 本公司需投入的人员配置
  • 每个周期要花多少钱
  • 多久回本 / 年度 ROI
  • 不含技术选型(老板不关心)

乙方版 看施工方案

目标:作为你报价和交付的施工图,要细。

  • 每周任务拆解(排期表)
  • 技术选型(模型 / 框架 / 工具)
  • Agent / 智能体方案设计
  • Token 成本拆解
  • 项目报价区间与毛利估算
本质

老板关心「投入产出和节奏」,乙方关心「怎么施工」。同一个项目计划,两种视角切出来。这两份报告共用一套计算内核,只是展示层不同——开发上不增加计算逻辑,只多一套渲染模板。

07 / TECH

技术建议

MVP 越轻越好。能不做后端就不做,目标是 2–3 周出真东西。

模块MVP 方案理由
前端纯静态网页(可上 Vercel / 国内静态托管)无需登录,一个链接就能发给客户
计算全部前端 JS 完成,无后端公式不复杂,前端算完即显示,零服务器成本
报告生成页面内渲染 + 浏览器打印 / 导出 PDF避免做后端文档服务,先验证
留资表单提交到飞书多维表格 / 问卷 / 邮件不建账号体系,人工跟进即可
埋点简单事件统计(填写完成、报告查看、留资、即将上线点击)这是验证假设的数据来源
后台系数配置MVP 可先写死在代码常量里等需要频繁改再做配置界面
关键取舍

MVP 不做账号、不做真实 API 对接执行、不做多场景配置器。整个工具本质是「一个会算账、会出报告的智能表单」。把复杂度全部留给后续真实交付项目,工具本身保持极简。

08 / METRICS

验证指标

MVP 上线后只盯这几个数,它们直接回答「核心假设成不成立」。

填写完成率
≥ 40%
进入录入页的人中,完整填完的比例。低于此说明字段太多或门槛太高。
报告查看率
≥ 80%
填完的人中查看完整报告的比例。验证「报告本身有没有吸引力」。
留资转化率
≥ 15%
看完报告后留下联系方式的比例。这是「能不能成交」最直接的信号。
场景需求分布
—
三个「即将上线」场景各自被点击 / 留资的次数,决定下一个做哪个。
北极星指标

对这个 MVP 来说,唯一真正重要的是:有没有拿到第一个愿意为落地服务付费的客户。上面的漏斗指标都是为这个最终目标服务的过程指标。

09 / SCOPE

范围边界

明确「不做什么」和「成功后做什么」,防止 MVP 范围蔓延。

MVP 明确不做

  • 完全自定义场景 / 自定义公式(让企业自己配)——这是未来,绝不在 MVP
  • 四个场景都做深——只做透电商内容,其余做入口
  • 账号 / 登录 / 权限体系
  • 真实 AI 接口对接与执行(MVP 只测算,不真的生成内容)
  • 后台系数可视化配置界面(先写死在代码里)

验证成功后的下一步

  1. 根据「即将上线」点击数据,做透需求第二大的场景
  2. 沉淀第一个真实交付案例,放到落地页做背书
  3. 把人工跟进的报价 / 交付流程标准化为可复用 SOP
  4. 开始积累「行业 ROI 数据库」——你真正的长期壁垒
