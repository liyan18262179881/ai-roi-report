# CLAUDE.md — AI 赋能商业计划报告生成器

> 本文件是项目的"大脑"。任何接手开发的人(或 Claude Code)先读这里,即可掌握全部背景、已定决策、计算逻辑、数据来源与待办,无需翻聊天记录。

---

## 0. 一句话定位

一个**面向中小电商企业的自助式网页工具**:用户填公司信息 + 选 AI 赋能方向 + 填几个真实数字,一键生成一份《AI 赋能 XX 品牌商业计划报告》(投资备忘录风格、可导出 PDF)。报告同时算清 **收益(降本+增效)** 与 **投入(自建 vs 外包)**,得出真实 ROI 与回本周期。

## 1. 商业背景(为什么做这个)

- **做的人**:AI 产品经理(独立/小团队),从零起步。
- **商业模式**:**工具 + 服务打包** —— 先用工具给客户做免费/低价评估,用一份专业报告建立信任,再承接后续 AI 落地实施服务。工具本身也可作为产品对外卖,或作为获客/谈单的钩子。
- **目标客户**:中小电商企业(约 50–500 人,没有 AI 团队)。
- **核心假设(MVP 要验证的)**:一份针对企业真实情况、数字可信的 ROI/商业计划报告,能不能打动企业、促成它为后续 AI 落地服务付费。验证的是"质量能否成交",不是"哪个场景热门"。
- **长期壁垒**:每做一个项目沉淀真实"行业 ROI 数据库" + 实施方法论 SOP。工具可被复制,行业数据与交付经验不能。

## 2. 产品形态与核心流程

形态:**纯前端单页网页**,无需注册登录,一个链接发给客户即可。

流程:左侧填【公司信息 + 勾选赋能方向】→ 中间填【各方向参数】→ 点【生成商业计划报告】→ 出报告(可【导出 PDF】/【返回编辑】)。

## 3. 已确认的关键产品决策(不要轻易推翻)

| # | 决策 | 理由 |
|---|------|------|
| D1 | 双视角:老板看回报、乙方/自己看部署投入 | 老板版促成交,乙方版定价保利润 |
| D2 | 多团队结构,5 个方向:视觉/设计、客服、销售/运营、内部知识库、HR | 电商是多部门的,每个方向 AI 赋能方式、算法、token 都不同 |
| D3 | 视觉价值给"降本 + 增效"两个口径 | 控成本的老板看降本,扩张期不想裁员的老板看产能翻倍 |
| D4 | **不编造数字**:空字段标「待补充」,且不计入测算合计 | 用户明确要求,可信度命脉 |
| D5 | 提效系数(产能倍数/AI处理率/增收幅度/省时/省事务比)**藏后台**,按 2026 行业基准取值,报告里注明 | 用户填不出这些系数,放前台是添乱 |
| D6 | 业务量(出图量/对话量/内容量)**用户直接填**,单价摆明面,价格 = 业务量 × 单价 | token 成本必须透明可追溯,否则客户质疑"价格哪来的" |
| D7 | 投入侧给 **自建 vs 外包** 两套成本对比 + 各自 ROI/回本 | 一份合格商业报告必须同时有"投入"和"收益"两面 |
| D8 | 自建团队规模**随方向数缩放**(1-2 场景轻配、4+ 场景重配) | 否则单场景自建显得贵得离谱 |
| D9 | 报告风格:简约黑白投资备忘录,可导出 PDF(html2pdf,不走打印对话框) | 用户偏好,易打印、像正式交付物 |
| D10 | **不自动拉取企查查/公开数据,不伪造真实公司财务** | 无后端 + 伪造客户财务会砸招牌。公司信息由用户填,标"基于输入估算";企查查留作未来后端功能 |
| D11 | 数据基准全部为 **2026 年 6 月** 口径 | 见第 5 节 |

## 4. 五个赋能方向的计算逻辑

通用:`hourly = 月薪/22/8`;人力成本统一 ×1.3(含社保公积金等综合用工成本)。
一个方向只有其 3 个用户字段都填了才"完整(complete)",才计入合计;否则标「待补充」。

### 4.1 视觉/设计(降本 + 增效)
- 用户字段:团队人数 n、人均月薪 sal、月出图量 vol
- 后台系数:产能提升 boost = 2.5
- 降本:`可精简人数 people = n × (1 − 1/boost)`;`月省 value = people × sal × 1.3`
- 增效:`新产能 = vol × boost`;`等效新增 = vol × (boost − 1)`
- API:`vol × 0.10 元/件`

### 4.2 客服(降本)
- 用户字段:团队人数 n、人均月薪 sal、月对话量 vol
- 后台系数:AI 自动处理 auto = 45%(2026 现实 20–30%、标杆 60%+ 的稳健取值)
- `people = n × (auto/100) × 0.7`(0.7 = 人工兜底折扣);`value = people × sal × 1.3`
- API:`vol × 0.012 元/次`

### 4.3 销售/运营(增收)
- 用户字段:月营收(万)rev、毛利率 margin%、月生成内容 vol
- 后台系数:AI 增收幅度 up = 12%(2026 区间 8–15%)
- `月增营收 extraRev = rev×10000 × up%`;`净价值 value = extraRev × margin%`
- API:`vol × 0.006 元/条`

### 4.4 内部知识库(降本)
- 用户字段:覆盖员工 n、人均月薪 sal、日均查询/人 q
- 后台系数:每次省时 save = 6 分钟
- `月查询量 vol = n × q × 22`;`value = vol × (save/60) × hourly`;`等效人力 people = value/(sal×1.3)`
- API:`vol × 0.01 元/次查询`

### 4.5 HR/人事(降本)
- 用户字段:HR 人数 n、人均月薪 sal、月简历/事务量 vol
- 后台系数:事务可省 save = 40%
- `people = n × save%`;`value = people × sal × 1.3`
- API:`vol × 0.01 元/件`

### 4.6 投入侧(自建 vs 外包)
设已完整方向数 = cc。
**自建团队**(2026 薪资):AI 产品经理 30000、AI 应用工程师 28000、数据/知识工程 15000、设计(选了视觉才有)10000。
- 缩放:工程师 `cc≥4?2:1`;PM `cc≥3?1:0.5`;数据 `(选了kb/cs/hr)?1:0.5`;建设月数 `min(6, 2+ceil(cc/2))`
- `建设期成本 = 团队月人力 × 建设月数`
- `运维人力/月 = cc≤1?15000 : cc≤3?25000 : 40000`
- `自建首年 = 建设期成本 + 运维人力×(12−建设月数) + API×12`
- `自建回本 = 建设期成本 / (月收益 − 运维人力 − API)`

**外包**:
- `一次性 = Σ(各方向 build 的 pre+mid)`(各方向 build 见 index.html 的 SC 对象)
- `运维托管/月 = round((API + 月收益×0.05) × 1.3)`
- `外包首年 = 一次性 + 托管×12`;`外包回本 = 一次性 / (月收益 − 托管)`

ROI:`(月收益×12 − 首年投入) / 首年投入 × 100%`。

> **重要口径注**:降本价值是"等效人力成本",只有实际减员或停止增招才转化为现金;若保留团队转去扩产,价值体现为产能增量而非现金。报告已在收益章加此注脚。

## 5. 2026 数据来源与基准(全部需定期复核)

- **大模型 API 价格**:2026 较 2025 暴跌约 80%。国产模型(DeepSeek/豆包/通义 Qwen)为主。客服对话单价按 Qwen-Plus 级估 ≈ 0.012 元/次。
- **token/调用单价(可在"我的视角/部署"里调)**:图 0.10 元/件、客服对话 0.012 元/次、运营内容 0.006 元/条、知识库查询 0.01 元/次。
- **岗位薪资(电商,一线/新一线)**:视觉设计 8000–9000(AI 冲击下增长停滞)、客服 5500、运营平均约 9200、知识库覆盖员工示例 10000、HR 8000。
- **AI 提效基准**:视觉产能 ×2–3 倍;客服独立接待率现实 20–30%、标杆 60%+、期望 60–70%,可处理约 80% 标准化问题;销售 AI 增收 8–15%。
- **自建岗位薪资 2026**:AI 产品经理普遍 3–5 万(中小企业应用岗约 3 万)、AI/大模型工程师 4–5 万(应用层约 2.8–3 万)、AI 数据训练师中位约 8500。

> 这些是公开行业基准,非某具体品牌真实数据。复核时用 web 搜索更新,并同步更新 index.html 里 SC 对象的 coef/price 与 compute() 里 selfBuild 的薪资常量。

## 6. 技术实现

- 单个 `index.html`,纯前端,无后端、无构建步骤。直接浏览器打开即用。
- PDF 导出:html2pdf.js(CDN: cdnjs)。注意:在 Claude.ai 的 artifact 沙盒里不能用 localStorage;独立部署无此限制。
- 字体:Source Serif 4(标题)+ JetBrains Mono(数字)+ 系统中文 sans。
- 代码结构(index.html `<script>` 内):
  - `SC` —— 五个方向的定义(名称/系数 coef/部署 deploy/模型 model/单价 price/一次性 build/6周排期 weeks/前台字段 params)。**改系数、改单价、改排期都在这里。**
  - `compute()` —— 所有计算 + 自建/外包投入模型。
  - `generate()` —— 渲染报告 HTML(9 个章节)。
  - `val(id)` —— 取值;空返回 null(这是"不编造"的关键,别改回带默认值)。
- 报告章节:01 执行摘要 / 02 现状分析 / 03 收益测算(含视觉双口径)/ 04 投入:自建vs外包 / 05 战略架构 / 06 人员与组织 / 07 路线图 / 08 风险 / 09 结论行动。

## 7. 当前状态

1. PDF 长表格断页优化(避免表格被从中间切断)+ 页脚更明确的免责声明。(此前被推迟的"第 4 项")

3. 留资 / 线索采集入口(报告末尾留联系方式),+ 简单埋点(填写完成率、报告查看率、留资率)用于验证核心假设。
4. （需后端)企查查/工商公开信息接入,自动带出公司画像。

**明确不做(防止范围蔓延)**:完全自定义场景/公式、账号体系、真实 AI 接口对接执行、伪造客户财务数据。

## 8. 给接手者的提醒

- 改系数/单价/薪资 → 同时改 `SC` 与 `compute()` 里 selfBuild 常量,并更新本文件第 5 节。
- 别破坏"不编造"原则:`val()` 必须保持空→null;新加字段也要进 `待补充` 逻辑。
- 所有数字都是"基于输入 + 行业基准的估算",报告页脚和注脚已声明,新增内容请沿用此口径,不要写成承诺。
- 数据每隔几个月用 web 搜索复核一次(API 降价快、薪资会变)。
